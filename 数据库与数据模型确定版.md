# 医学数据库管理软件 - 数据库表结构与核心数据模型（确定版）

> **本文件为最终确定版**，所有表结构、字段定义、Rust数据模型以此为准。  
> 编写日期: 2026-02-26  
> 技术栈: Rust + Actix-web 4 + PostgreSQL 15+ + Diesel 2 + MinIO  

---

## 1. 设计原则

1. **先做存储管理**：Phase 1 只实现核心业务表（记录 + 文件 + 字典），用户/权限/审计留到 Phase 2
2. **部位驱动存储**：每个身体部位对应一个 MinIO 桶，桶内按业务类型（images/models/annotations/reports）组织
3. **外键关联字典表**：医院和部位使用独立字典表 + 外键，保证数据标准化
4. **Diesel ORM**：所有表对应 Rust Queryable/Insertable 结构体，编译期类型安全

## 3. Phase 1 表结构详细定义

### 3.1 身体部位字典表 (body_parts)

每个部位对应一个 MinIO 桶。新增部位时程序自动创建对应桶。

```sql
CREATE TABLE body_parts (
    id              SERIAL PRIMARY KEY,
    part_code       VARCHAR(50) UNIQUE NOT NULL,     -- 部位编码（英文，如 knee）
    part_name       VARCHAR(100) NOT NULL,           -- 部位中文名称（如 膝关节）
    bucket_name     VARCHAR(100) UNIQUE NOT NULL,    -- MinIO桶名称（如 medical-knee）
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,   -- 是否启用
    sort_order      INTEGER NOT NULL DEFAULT 0,      -- 排序顺序
    description     TEXT NOT NULL DEFAULT '',         -- 描述
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_body_parts_active ON body_parts(is_active);
```

### 3.3 医学数据记录主表 (medical_records)

每条记录代表一次医学数据采集，关联一个医院和一个部位。文件通过 `files` 表关联。

```sql
CREATE TABLE medical_records (
    id              SERIAL PRIMARY KEY,
    record_no       VARCHAR(50) UNIQUE NOT NULL,     -- 记录编号（自动生成：MR-2026-000001）
    year            INTEGER NOT NULL,                -- 年度
    month           SMALLINT NOT NULL CHECK (month BETWEEN 1 AND 12),
    day             SMALLINT NOT NULL CHECK (day BETWEEN 1 AND 31),
    record_date     DATE NOT NULL,                   -- 完整日期（year+month+day，便于范围查询）
    hospital_id     INTEGER NOT NULL REFERENCES hospitals(id),    -- 医院（外键）
    body_part_id    INTEGER NOT NULL REFERENCES body_parts(id),   -- 部位（外键 → 决定MinIO桶）
    description     TEXT NOT NULL DEFAULT '',         -- 描述/备注
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_records_date ON medical_records(record_date);
CREATE INDEX idx_records_hospital ON medical_records(hospital_id);
CREATE INDEX idx_records_body_part ON medical_records(body_part_id);
CREATE INDEX idx_records_year_month ON medical_records(year, month);
CREATE INDEX idx_records_record_no ON medical_records(record_no);
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 自增主键 |
| `record_no` | VARCHAR(50) | 业务编号，格式 `MR-{年}-{6位序号}`，如 `MR-2026-000001` |
| `year` | INTEGER | 数据采集年份 |
| `month` | SMALLINT | 数据采集月份（1-12） |
| `day` | SMALLINT | 数据采集日期（1-31） |
| `record_date` | DATE | `year-month-day` 组合的完整日期，用于范围查询 |
| `hospital_id` | INTEGER FK | 关联 hospitals 表 |
| `body_part_id` | INTEGER FK | 关联 body_parts 表。**核心**：通过部位确定 MinIO 桶 |
| `description` | TEXT | 自由文本描述 |
| `created_at` | TIMESTAMP | 记录创建时间 |
| `updated_at` | TIMESTAMP | 记录最后更新时间 |

### 3.4 文件元数据表 (files)

每个文件对应一条记录，存储 MinIO 中的定位信息。

```sql
CREATE TABLE files (
    id              SERIAL PRIMARY KEY,
    record_id       INTEGER NOT NULL REFERENCES medical_records(id) ON DELETE CASCADE,
    file_type       VARCHAR(20) NOT NULL,            -- 业务类型：image / model / annotation / report
    original_name   VARCHAR(500) NOT NULL,           -- 原始文件名
    storage_name    VARCHAR(500) NOT NULL,           -- 存储文件名（UUID.ext）
    bucket_name     VARCHAR(100) NOT NULL,           -- MinIO桶名（冗余，来自 body_parts.bucket_name）
    object_key      VARCHAR(1000) NOT NULL,          -- MinIO对象键（{业务类型}/{年}/{月}/{记录ID}/{UUID}.{ext}）
    file_size       BIGINT NOT NULL DEFAULT 0,       -- 文件大小（字节）
    file_extension  VARCHAR(20) NOT NULL DEFAULT '', -- 文件扩展名（dcm / nii.gz / stl / zip / pdf / docx / xlsx / md）
    upload_time     TIMESTAMP NOT NULL DEFAULT NOW() -- 上传时间
);

CREATE INDEX idx_files_record_id ON files(record_id);
CREATE INDEX idx_files_file_type ON files(file_type);
CREATE INDEX idx_files_bucket ON files(bucket_name);
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 自增主键 |
| `record_id` | INTEGER FK | 关联 medical_records，级联删除 |
| `file_type` | VARCHAR(20) | 业务分类，值域：`image`(医学图像) / `model`(模型) / `annotation`(标注) / `report`(报告) |
| `original_name` | VARCHAR(500) | 用户上传时的原始文件名 |
| `storage_name` | VARCHAR(500) | MinIO中的文件名（UUID重命名，如 `a1b2c3d4.dcm`） |
| `bucket_name` | VARCHAR(100) | MinIO桶名（冗余存储，避免每次查部位表。如 `medical-knee`） |
| `object_key` | VARCHAR(1000) | MinIO对象键，格式：`{file_type}/{year}/{month}/{record_id}/{storage_name}` |
| `file_size` | BIGINT | 文件大小，字节 |
| `file_extension` | VARCHAR(20) | 扩展名，支持双扩展名如 `nii.gz` |
| `upload_time` | TIMESTAMP | 上传时间 |

**file_type 与文件格式的对应关系**：

| file_type | 业务分类 | 允许的文件格式 |
|-----------|---------|---------------|
| `image` | 医学图像数据 | .dcm, .nii.gz |
| `model` | 模型数据 | .stl |
| `annotation` | 标注数据 | .zip, .nii.gz |
| `report` | 报告 | .pdf, .docx, .xlsx, .md |

> `.nii.gz` 同时可归属 image 和 annotation，由用户在前端选择上传到哪个业务分类。

**Rust 结构体**：

```rust

    pub storage_name: &'a str,
    pub bucket_name: &'a str,
    pub object_key: &'a str,
    pub file_size: i64,
    pub file_extension: &'a str,

// 列表展示用（含业务类型中文名）
impl FileRecord {
    pub fn file_type_display(&self) -> &str {
        match self.file_type.as_str() {
            "image" => "医学图像",
            "model" => "模型",
            "annotation" => "标注",
            "report" => "报告",
            _ => "未知",
        }
    }
    
    pub fn file_size_display(&self) -> String {
        if self.file_size < 1024 {
            format!("{} B", self.file_size)
        } else if self.file_size < 1024 * 1024 {
            format!("{:.1} KB", self.file_size as f64 / 1024.0)
        } else if self.file_size < 1024 * 1024 * 1024 {
            format!("{:.1} MB", self.file_size as f64 / (1024.0 * 1024.0))
        } else {
            format!("{:.2} GB", self.file_size as f64 / (1024.0 * 1024.0 * 1024.0))
        }
    }
}
```

## 5. MinIO 存储结构（确定版）

### 5.1 桶设计：按部位分桶

| 桶名称 | 部位 | 对应 body_parts.bucket_name |
|--------|------|-----------------------------|
| `medical-scapula` | 肩胛骨 | `medical-scapula` |
| `medical-elbow` | 肘关节 | `medical-elbow` |
| `medical-pelvis` | 骨盆 | `medical-pelvis` |
| `medical-knee` | 膝关节 | `medical-knee` |
| `medical-ankle` | 足踝 | `medical-ankle` |
| `medical-hand` | 手掌 | `medical-hand` |

### 5.2 桶内目录结构：按业务类型 + 时间

```
{桶名}/
├── images/           ← 医学图像（.dcm, .nii.gz）
│   └── {年}/{月}/{记录ID}/{uuid}.{ext}
├── models/           ← 模型数据（.stl）
│   └── {年}/{月}/{记录ID}/{uuid}.{ext}
├── annotations/      ← 标注数据（.zip, .nii.gz）
│   └── {年}/{月}/{记录ID}/{uuid}.{ext}
└── reports/          ← 报告（.pdf, .docx, .xlsx, .md）
    └── {年}/{月}/{记录ID}/{uuid}.{ext}
```

### 5.3 Object Key 规则

```
格式: {file_type}/{year}/{month}/{record_id}/{uuid}.{extension}

示例（medical-knee 桶内）：
images/2026/01/42/a1b2c3d4-e5f6-7890-abcd-ef1234567890.dcm
images/2026/01/42/b2c3d4e5-f6a7-8901-bcde-f12345678901.nii.gz
models/2026/01/42/c3d4e5f6-a7b8-9012-cdef-123456789012.stl
annotations/2026/02/42/d4e5f6a7-b8c9-0123-defa-234567890123.zip
annotations/2026/02/42/f6a7b8c9-d0e1-2345-fgab-456789012345.nii.gz
reports/2026/01/42/e5f6a7b8-c9d0-1234-efab-345678901234.pdf
```

### 5.4 文件定位完整路径

```
bucket(来自 body_parts.bucket_name) + object_key(来自 files.object_key) = 唯一文件定位

例：medical-knee / images/2026/01/42/a1b2c3d4.dcm
```

- `files` → `medical_records`：多对一，级联删除（删除记录时自动删除关联文件记录）

```sql
    day             SMALLINT NOT NULL CHECK (day BETWEEN 1 AND 31),
    record_date     DATE NOT NULL,
    hospital_id     INTEGER NOT NULL REFERENCES hospitals(id),
    body_part_id    INTEGER NOT NULL REFERENCES body_parts(id),
    description     TEXT NOT NULL DEFAULT '',
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_records_date ON medical_records(record_date);
CREATE INDEX idx_records_hospital ON medical_records(hospital_id);
CREATE INDEX idx_records_body_part ON medical_records(body_part_id);
CREATE INDEX idx_records_year_month ON medical_records(year, month);

CREATE TRIGGER update_medical_records_updated_at
    BEFORE UPDATE ON medical_records
    FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

-- ============================
-- 4. 文件元数据表
-- ============================
CREATE TABLE files (
    id              SERIAL PRIMARY KEY,
    record_id       INTEGER NOT NULL REFERENCES medical_records(id) ON DELETE CASCADE,
    file_type       VARCHAR(20) NOT NULL CHECK (file_type IN ('image', 'model', 'annotation', 'report')),
    original_name   VARCHAR(500) NOT NULL,
    storage_name    VARCHAR(500) NOT NULL,
    bucket_name     VARCHAR(100) NOT NULL,
    object_key      VARCHAR(1000) NOT NULL,
    file_size       BIGINT NOT NULL DEFAULT 0,
    file_extension  VARCHAR(20) NOT NULL DEFAULT '',
    upload_time     TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_files_record_id ON files(record_id);
CREATE INDEX idx_files_file_type ON files(file_type);
CREATE INDEX idx_files_bucket ON files(bucket_name);
```

## 8. 核心业务流程中的数据流

### 8.1 新建记录 + 上传文件

```
用户填写表单（选择医院、部位、日期、文件）
    │
    ▼
后端接收请求
    │
    ├─→ 从 body_parts 表查 bucket_name（如 "medical-knee"）
    ├─→ 生成 record_no（如 "MR-2026-000001"）
    ├─→ INSERT INTO medical_records（获得 record.id）
    │
    ▼
处理每个上传文件
    │
    ├─→ 获取 file_type（前端传入：image/model/annotation/report）
    ├─→ UUID 重命名 → storage_name
    ├─→ 构建 object_key = "{file_type}/{year}/{month}/{record.id}/{storage_name}"
    ├─→ 上传到 MinIO: bucket_name + object_key
    └─→ INSERT INTO files（记录MinIO定位信息）
```

### 8.2 下载文件

```
用户点击下载
    │
    ▼
后端查询 files 表 → 获得 bucket_name + object_key
    │
    ├─→ MinIO presigned URL（推荐）→ 302重定向
    └─→ 或代理下载 → 流式返回
```

### 8.3 删除记录

```
用户点击删除记录
    │
    ▼
后端查询该记录的所有 files → 逐个从 MinIO 删除对象
    │
    ▼
DELETE FROM medical_records WHERE id = ?
    └─→ files 表因 ON DELETE CASCADE 自动清理
```

## 9. 与旧文档的差异说明

| 项目 | 旧数据库设计文档 | 本确定版 | 原因 |
|------|-----------------|---------|------|
| medical_records 字段数 | 30+ 字段 | 11 字段 | Phase 1 精简，患者信息/影像类型等后续扩展 |
| 医院 | FK → hospitals（含地址/联系人等） | FK → hospitals（精简） | 当前只需名称，后续可加字段 |
| 部位 | FK → body_parts（含层级/分类） | FK → body_parts（+bucket_name） | 新增 bucket_name 关联 MinIO |
| 文件存储路径 | medical_records 内嵌路径字段 | 独立 files 表 | 一条记录可有多个文件，1:N更合理 |
| 文件表 | file_metadata（含hash/加密） | files（精简） | Phase 1 不需要hash/加密 |
| 用户/权限 | 完整 RBAC（5张表） | Phase 2 | 先做存储管理，后加权限 |
| 审计日志 | audit_logs | Phase 2 | 同上 |
| MinIO桶 | 无 | 按部位6个桶 | 新架构决策 |
